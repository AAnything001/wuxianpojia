# 官网首页免费模块栏目设计

## 目标

在官网首页现有痛点卡片栏目下方新增“免费模块”栏目，向首次访问者清楚说明 Windows 新版 EXE 内置的免费环境强化能力，并把主要行为集中到下载 EXE。

栏目采用已确认的方案 C“免费与专业分层”：免费模块是视觉主角，专业破甲只用于说明产品边界。此次只修改官网表现和下载入口，不改 EXE、CDK、激活、付费 payload、MCP 或工具链逻辑。

## 位置与页面结构

- 修改 `index.html` 的默认 A 版页面。
- 新栏目插入当前痛点卡片与“了解破甲”横条之后、`#a-cap` 专业能力栏目之前。
- 栏目 ID 为 `a-free-modules`，供测试和以后站内链接使用；本次不增加顶部导航项，避免挤压现有导航。
- 栏目使用现有 `.section` 宽度、背景网格、字体和红紫品牌色，不改变后续价格、下载和页脚结构。

## 免费与专业层级

桌面端采用不对称双栏：免费卡片约占 70%，专业卡片约占 30%。移动端改为单列，免费卡片始终在前。

### 免费卡片

- 标题：`免费环境强化`
- 徽标：`无需 CDK`
- 主文案：`11 个常用 Skill，下载即可选装。`
- 说明：`覆盖设计、创意、内容、研究和能力构建，模块已经随 EXE 内置。`
- 三个数字：`11 免费模块`、`5 能力分类`、`6 AI 客户端`
- 五个分类：`设计`、`创意`、`内容与演示`、`研究`、`能力构建`
- 客户端说明：`自动检测并适配 Claude、Codex、Kiro、Cursor、Cline、Windsurf`
- 主按钮：`免费下载 Windows EXE`
- 按钮辅助文字：`模块已内置 · 无需运行时下载`

主按钮复用首页现有 `data-download-placeholder` 机制，点击后打开已有下载面板，不新增下载 URL 或第二套弹窗。

### 专业卡片

- 标题：`专业破甲能力`
- 徽标：`激活后使用`
- 主文案：`需要深入分析时，再解锁完整工作流。`
- 边界说明：`免费模块不会激活或替代专业逆向能力。`
- 能力摘要：`完整 Reverse Skill`、`MCP 与工具链配置`、`授权软件与 CTF 逆向流程`、`配置备份、校验与恢复`
- 弱化链接：`了解专业破甲 →`，跳转到 `codex-pojia.html`

专业卡片不得出现购买价格、CDK 输入、购买按钮或与免费主按钮竞争的高亮样式。

## 产品事实与边界

免费模块事实以 Windows v1.30.1 已确认设计为准：

- 11 个 Skill 随 EXE 内置，用户无需激活 CDK 即可选择安装。
- 免费能力面向设计、创意、内容与演示、研究、能力构建五类用途。
- 支持程序自动检测 Claude、Codex、Kiro、Cursor、Cline、Windsurf。
- 免费流程不部署完整 reverse-skill，不配置 MCP，不下载或启动第三方工具。
- 免费流程不读取或改变 CDK 状态，也不解锁付费功能。

官网不使用“破解免费版”“免费解锁专业能力”等可能混淆付费边界的表达。

## 视觉与响应式

- 免费卡片使用轻量红色边框、柔和品牌光晕和高对比下载按钮，承担主视觉。
- 专业卡片使用中性白灰配色和普通文字链接，承担解释作用。
- 使用有限的圆角和分隔线，不把每条文字都做成独立卡片。
- 桌面端保持不对称双栏；宽度小于 `850px` 时切换单列。
- 移动端主标题、数字和按钮不得溢出；下载按钮允许辅助文字换行。
- 遵守 `prefers-reduced-motion`，本栏目不新增持续动画。

## 交互与失败处理

- 点击“免费下载 Windows EXE”必须调用现有下载弹窗并把焦点移到关闭按钮。
- 下载弹窗关闭后，焦点返回免费模块下载按钮。
- 点击“了解专业破甲”正常打开站内专题页。
- 即使 JavaScript 不可用，专业专题链接仍可访问；下载按钮保持现有首页的 JavaScript 行为，不新建降级逻辑。

## 验证标准

- 栏目准确出现在痛点栏目之后、`#a-cap` 之前。
- 页面只出现一个 `#a-free-modules`。
- 11、5、6 三个数字和六个客户端名称全部可见。
- 免费与付费边界文案准确，不把免费模块描述为完整破甲能力。
- Playwright 在 `1280×900` 和 `390×844` 视口下无横向溢出。
- Playwright 点击免费按钮后，现有下载弹窗变为可见；关闭后焦点返回原按钮。
- “了解专业破甲”链接指向 `codex-pojia.html`。
- 现有购买链接、下载 URL、用户须知入口、更新日志、CDK 和激活相关内容没有差异。
- 仓库现有测试、HTML 静态检查和 `git diff --check` 通过。
- 实现完成并通过自动化检查后，启动本地静态测试服务，保持服务运行，并把可直接访问的完整测试 URL 提供给用户验收。
- 本轮先停在本地测试 URL 供用户验收，不提前推送；用户确认预览后，再只提交本任务文件并推送 `origin/main`，随后验证 Cloudflare 线上栏目、弹窗交互和移动端布局。
